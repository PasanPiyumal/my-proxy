Building this SOCKS5 proxy required focused learning on the handshake sequence in RFC 1928 and username/password negotiation in RFC 1929. The most important part to learn quickly was packet framing: each stage has variable-length fields, so the server must safely buffer partial chunks and only parse when enough bytes have arrived. I also had to understand where a minimal implementation can stop while still being useful, which is why this project supports `CONNECT` tunneling only.

For debugging, I used an iterative approach: verify greeting negotiation first, then authentication, then destination parsing, then upstream connection, and finally full duplex piping. I treated each stage as a state machine and validated behavior by checking expected byte-level responses and logging clear connection details. This made it easier to isolate failures, such as unsupported auth methods, malformed address types, or connection errors from upstream targets.

With more time, I would add automated integration tests (for handshake/auth/request cases), stronger IPv6 bound-address encoding, and optional structured logging output for easier ingestion by monitoring systems. I would also add configurable access policies (allowlist/denylist for hosts and ports), rate limits, and graceful shutdown behavior to make the proxy more production-ready while keeping the implementation readable.
